# Checkpoint resume control API

Add script-level and checkpoint-level controls for resume behavior.

## Problem

Current checkpoint behavior is implicit - scripts resume from checkpoints by default with no way to declare intent. This causes confusion:
- User-facing scripts should default to fresh runs but currently cache unexpectedly
- ETL scripts should resume by default but currently require manual cache management
- No way to express 'this checkpoint should never be cached' or 'only cache for 1 hour'

Real example from Beane project: User asked 'which players should I drop?' but got cached strategy from previous 'sleeper holds' question because checkpoints persisted across unrelated queries.

## Proposed API

### Script-level directive (top of .mld file)

```mlld
resume: never    >> Never use checkpoints (good for user-facing analysis)
resume: auto     >> Use checkpoints by default (good for ETL)
resume: manual   >> Only use checkpoints with --resume flag
```

Supports both quoted and unquoted: `resume: never` or `resume: "never"`

Default if not specified: `resume: auto` (current behavior)

### Per-checkpoint controls

```mlld
checkpoint "strategy" with {
  resume: manual,              >> Override script-level (only resume with --resume=strategy)
  ttl: 1h,                     >> Auto-expire after 1 hour (also: 30m, 24h, 7d)
  complete: @strategy.done     >> Mark complete when condition true
}
```

**Checkpoint-level `resume:` overrides script-level.**

### Completion detection

If `complete:` condition is true, skip checkpoint even in resume mode:

```mlld
checkpoint "sync" with {
  complete: @data.synced == true
}

var @data = @fetchData()
if @data.synced [
  => @data  >> Early return, checkpoint marked complete
]
```

On next run with `resume: auto`, this checkpoint is skipped because `@data.synced == true`.

### TTL parsing

Support human-readable durations:
- `ttl: 5m` → 5 minutes
- `ttl: 1h` → 1 hour  
- `ttl: 24h` or `ttl: 1d` → 1 day
- `ttl: 7d` → 1 week

## Use Cases

**User-facing analysis tool (Beane):**
```mlld
resume: never    >> Always fresh, never cache between queries
```

**Data pipeline with slow source:**
```mlld
resume: auto

checkpoint "fetch_api" with {
  ttl: 1h,                    >> Don't refetch within 1h
  complete: @data.length > 0
}
```

**Development workflow:**
```mlld
resume: manual    >> Only resume when explicitly debugging

checkpoint "expensive_phase" with {
  resume: auto,   >> But this one can auto-resume (override)
  ttl: 24h
}
```

## Implementation Notes

- Script-level directive parsed before execution
- Checkpoint-level settings override script-level
- TTL checked on checkpoint load (ignore if expired)
- `complete:` evaluated after checkpoint block, cached for next run
- `--resume` flag overrides all settings (manual resume mode)
- `--resume=checkpoint_name` resumes from specific checkpoint

## Current Workaround

```mlld
exe @clearCheckpoints() = sh { rm -rf .mlld/checkpoints/ask }
var @cleared = @clearCheckpoints()
```

Works but hacky - need native support.
