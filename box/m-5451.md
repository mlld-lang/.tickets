# Phase 5: Configless box + VFS resolver shorthand

Spec: `_specs/box.md` (Part 4: Built-in box runtime)

Grammar changes for configless box blocks and VFS resolver shorthand. Evaluator creates VFS workspace from config. Sandbox defaults applied.

## New Grammar Forms

### box @resolver [...] (VFS resolver shorthand)
```mlld
box @workspace [
  run cmd { npm test }
]
```
Uses an existing resolver's VFS (created by `files <@workspace/>` in Phase 4) as the box's workspace. Shorthand for `box { fs: @workspace } [...]`.

### box { fs: @workspace } [...] (config form with VFS)
```mlld
box { fs: @workspace, tools: ["Bash", "Read"] } [
  run cmd { npm test }
]
```
The longhand form — specify VFS via `fs:` field in the config object, along with other config options. The `box @resolver` shorthand is syntactic sugar for this when no other config is needed.

### box [...] (anonymous VFS)
```mlld
box [
  file "task.md" = @task
  run cmd { cat task.md }
]
```
Creates an anonymous `VirtualFS.empty()` workspace. Useful when `file`/`files` inside the block populate it.

### Existing form unchanged
```mlld
box { tools: ["Bash"] } [...]
```
Config form without `fs:` field — no VFS workspace, just tool/MCP/network restrictions on local execution.

## Grammar Changes (grammar/directives/box.peggy)
```
BoxDirective
  = BoxKeyword config:BoxConfig WithClause? Block     // existing: box { ... } [...]
  / BoxKeyword resolver:ResolverRef WithClause? Block  // new: box @workspace [...]
  / BoxKeyword Block                                    // new: box [...] (anonymous VFS)
```

## Evaluator Changes (evaluateBox)

1. **Resolve config**: determine if config is object (existing), VFS resolver, or absent
2. **Create workspace**:
   - VFS resolver → use its VFS (from `files <@workspace/>`)
   - Config with `fs:` field → resolve fs value to VFS
   - Absent (anonymous) → create `VirtualFS.empty()`
   - No VFS config, no resolver → no workspace (existing behavior, restrictions only)
3. **Push workspace**: if workspace created, `childEnv.pushActiveWorkspace(workspaceValue)`
4. **Apply defaults** for VFS-backed boxes:
   - Network disabled (unless `net:` specified)
   - MCP disabled (unless `mcps:` specified)
   - Bash/Read/Write tools enabled (unless `tools:` restricts)
5. **Execute block**: commands route through ShellSession via Phase 2's CommandExecutorFactory
6. **Pop workspace**: in finally block (child env is destroyed anyway, but explicit is better)
7. **Return**: workspace state available for inspection on result

## Default Precedence (VFS mode) — spec Part 4

| Config | Default | Override |
|--------|---------|----------|
| tools | ["Bash", "Read", "Write"] | Parent can only restrict further, never expand |
| net | { allow: [] } (disabled) | Box config can allow; parent restrictions still apply |
| mcps | [] (disabled) | Box config can allow; parent restrictions still apply |
| auth | inherited from parent | Box config can add; parent auth carries through |
| provider | ShellSession (built-in) | Explicit provider overrides ShellSession |

## Escalation Path (spec Part 4)
```mlld
>> Level 0: no box (commands run on real FS, no restrictions)
run cmd { npm test }

>> Level 1: box with restrictions only (no VFS, no ShellSession)
box { tools: ["Bash", "Read"] } [
  run cmd { npm test }
]

>> Level 2: box with VFS (in-memory sandbox via ShellSession)
box @workspace [
  run cmd { npm test }
]

>> Level 3: box with Docker provider (full OS isolation)
box { provider: "@mlld/docker", fs: @workspace } [
  run cmd { npm test }
]
```

## Acceptance Criteria

- ShellSession created lazily on first command in box (via Phase 2 infrastructure)
- Existing `box { config } [...]` form unchanged (no VFS, restrictions only)
- Nesting works: `box @outer [ box @inner [...] ]` with proper stack push/pop
- Grammar tests for all forms (resolver shorthand, config+fs, anonymous, existing config)
- Evaluator tests for workspace creation, defaults, nesting
- Fixture tests for VFS-backed box execution
- Docs atoms updated for new box forms
- All existing tests pass
