# Phase 2: Workspace model + Environment integration

Spec: `_specs/box.md` (Part 2: Workspace model, Part 3: cleanup semantics)

Add workspace stack to Environment and integrate ShellSession routing into command execution. This wires the existing VirtualFS + ShellSession (`services/fs/`) into the interpreter.

## Existing Infrastructure
- **VirtualFS** (`services/fs/VirtualFS.ts`): `VirtualFS.empty()`, `VirtualFS.over(backing)`, shadow-layer file operations, `changes()`, `diff()`, `fileDiff()`, `export()`/`apply()`
- **ShellSession** (`services/fs/ShellSession.ts`): `ShellSession.create(vfs, options?)`, `exec(commandLine)`, `getCwd()`, `getEnv()`
- **Environment** already accepts `IFileSystemService` (which VirtualFS implements) via constructor

## New Types

### WorkspaceValue (define in core/types/)
```typescript
interface WorkspaceValue {
  type: 'workspace';
  fs: VirtualFS;
  descriptions: Map<string, string>;  // path → description
  shellSession?: ShellSession;        // created lazily on first command
}
```

### WorkspaceProvider interface (define near CommandExecutorFactory)
```typescript
export interface WorkspaceProvider {
  getActiveWorkspace(): WorkspaceValue | undefined;
}
```

## Environment Changes (interpreter/env/Environment.ts)

Add workspace stack field and API:
```typescript
private workspaceStack: WorkspaceValue[] = [];

pushActiveWorkspace(workspace: WorkspaceValue): void {
  this.workspaceStack.push(workspace);
}

popActiveWorkspace(): WorkspaceValue | undefined {
  return this.workspaceStack.pop();
}

getActiveWorkspace(): WorkspaceValue | undefined {
  return this.workspaceStack[this.workspaceStack.length - 1];
}
```

### Child environment inheritance
When `createChild()` is called, child COPIES parent's workspace stack (shallow copy of array):
- Commands inside box blocks see the box's workspace (exe invocations create child envs)
- Child can push/pop independently without affecting parent
- Parent and child stacks diverge after creation (no shared reference)

## CommandExecutorFactory Integration (interpreter/env/executors/CommandExecutorFactory.ts)

- Add `WorkspaceProvider` to `ExecutorDependencies` interface
- Environment implements `WorkspaceProvider`
- Wire in `Environment.getCommandExecutorFactory()`: pass `workspaceProvider: this`
- In `executeCommand()`: check `workspaceProvider.getActiveWorkspace()` at execution time (NOT factory creation time)
- If workspace exists and has ShellSession → route command through `shellSession.exec()`
- If workspace exists but no ShellSession → create lazily via `ShellSession.create(workspace.fs)`, store on `workspace.shellSession`, then route
- If no workspace → fall through to standard executors (ShellCommandExecutor, BashExecutor)

**Key timing:** Factory is lazy-created singleton per Environment. ActiveWorkspace changes dynamically between commands, so check MUST happen in `executeCommand()` at call time, not during factory creation.

**Provider routing (no change):** Provider-based execution routes through `executeProviderCommand()` at three call sites (`run-exec-definition-dispatcher.ts:354`, `run-command-executor.ts:409`, `execute-command.ts:124`) and bypasses CommandExecutorFactory entirely. ShellSession integration targets local execution only.

## executeInWorkingDirectory() Helper (interpreter/utils/working-directory.ts)

Create helper with try/finally for consistent workspace stack semantics:
```typescript
export async function executeInWorkingDirectory<T>(
  workingDir: ContentNodeArray | string | undefined,
  env: Environment,
  execute: (resolvedPath?: string) => Promise<T>
): Promise<T> {
  const resolved = await resolveWorkingDirectory(workingDir, env);
  try {
    return await execute(resolved);
  } finally {
    // future: pop workspace if pushed (Phase 6 adds this)
  }
}
```

For Phase 2, this is a thin wrapper — the workspace-push/pop logic is added in Phase 6 when `resolveWorkingDirectory()` gains the `WorkingDirectoryResult` return type. The wrapper is introduced now so all call sites are migrated before the return type changes.

## Nesting behavior (spec Part 2)
```mlld
box @outer [
  >> commands use outer workspace's ShellSession
  run cmd { echo "outer" }

  box { fs: @inner } [
    >> commands use inner workspace's ShellSession (separate VFS)
    run cmd { cat file.md }
  ]

  >> back to outer workspace's ShellSession
  run cmd { echo "still outer" }
]
```
Child box blocks create child environments with a COPY of parent's workspace stack. Child pushes its own workspace in `evaluateBox()`. Parent stack unaffected.
