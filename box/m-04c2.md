# Phase 3: Unified write abstraction

Spec: `_specs/box.md` (Part 5 prerequisite, referenced in implementation order)

Consolidate all write paths into one `executeWrite()` function with policy/audit/routing. This is a prerequisite for file/files directives (Phase 4).

## Context

Write operations are currently scattered across evaluators. Phase 4 (file/files directives) needs to write to three different targets — real filesystem, VFS workspace, and resolver-backed VFS — through the same policy, audit, and routing infrastructure. This phase creates that unified abstraction.

**Note:** The spec references `spec-file-projection.md` for this abstraction, but that doc doesn't exist. Requirements are inferred from `_specs/box.md` Part 5 and the write target needs of Phases 4-8.

## Discovery Required

Before implementing, audit the current write paths to understand what needs consolidating:

### Current write evaluators to investigate
- `interpreter/eval/output.ts` — `output` directive (writes to file or stdout)
- `interpreter/eval/append.ts` — `append` directive (appends to file)
- Pipeline effect handlers that write output (in `interpreter/eval/pipeline/`)
- Any other file-writing code paths in the evaluator

### Questions to answer during discovery
- What policy checks exist today on writes? (capabilities, permissions)
- What audit logging exists today on writes?
- How do writes currently determine their target (filesystem path)?
- Where does the write actually happen? (direct `fs.writeFile()` calls vs abstracted)

## Scope

### executeWrite() function
Create a single write entry point that handles:

**Policy enforcement:**
- Check if write is allowed (capabilities, policy constraints)
- Honor environment-level restrictions (e.g., box with restricted tools)

**Target routing:**
- Real filesystem: existing behavior for `output`/`append` directives
- VFS workspace: when active workspace exists (`env.getActiveWorkspace()`), route writes to `workspace.fs.writeFile()`
- Resolver-backed VFS: for named `files <@resolver/>` targets (Phase 4 will use this)

**Audit logging:**
- Record what was written and where
- Include taint/label information if present

**Error handling:**
- Consistent error messages across all write paths
- Path validation

### Integration
- `output` evaluator uses executeWrite() instead of direct fs calls
- `append` evaluator uses executeWrite() instead of direct fs calls
- Pipeline effects that write use executeWrite()
- All existing behavior preserved (no user-visible changes)

## Acceptance Criteria
- Current write paths audited and documented (which files, what they do)
- executeWrite() function exists with policy/audit/routing
- Supports real filesystem writes (existing behavior)
- Supports VFS writes (writes to workspace.fs when active workspace present)
- Existing output/append behavior unchanged (all output/append tests pass)
- Pipeline effect writes route through new abstraction
- Unit tests for each write target type (filesystem, VFS)
- All existing tests pass
- Committed before proceeding to Phase 4
