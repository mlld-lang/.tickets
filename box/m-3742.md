# Phase 6: cmd:@workspace — workspace-aware exe definitions

Spec: `_specs/box.md` (Part 3: cmd:@workspace)

Pre-interpolation type checking in `resolveWorkingDirectory()` to detect WorkspaceValue before string interpolation. ShellSession activation for workspace targets.

## Syntax
```mlld
exe @lint(src) = cmd:@src { eslint . }
exe @test(ws) = cmd:@ws { npm test }
exe @claude(prompt, tools, ws) = cmd:@ws { claude -p "@prompt" --allowed-tools @tools }
```

## No Grammar Changes Needed
Existing `cmd:` parsing already supports variable interpolation via `@resolver` references through the `UnifiedVariableNoTail` pattern. The distinction happens at runtime, not parse time.

## Runtime Resolution Changes (interpreter/utils/working-directory.ts)

### Updated return type for resolveWorkingDirectory()
```typescript
export type WorkingDirectoryResult =
  | { type: 'path'; path: string; workspacePushed: false }
  | { type: 'workspace'; workspacePushed: true }
  | { type: 'none'; workspacePushed: false };
```

### Pre-interpolation type check
The key insight: `interpolate()` has a string-only contract (`Promise<string>`). If it encounters an object, it JSON-stringifies it. WorkspaceValue must be detected BEFORE interpolation.

Resolution order:
1. If `workingDir` is empty/absent → return `{ type: 'none' }`
2. If `workingDir` is a single variable reference node:
   a. Look up variable directly: `env.getVariable(varName)`
   b. If not found but exists as resolver: `await env.getResolverVariable(varName)`
   c. If value is a WorkspaceValue (`typeof value === 'object' && value.type === 'workspace'`):
      - Push onto workspace stack: `env.pushActiveWorkspace(value)`
      - Return `{ type: 'workspace', workspacePushed: true }`
3. Otherwise → interpolate to string → validate as filesystem path → return `{ type: 'path' }`

### Update executeInWorkingDirectory() helper (from Phase 2)
Add workspace pop to the finally block:
```typescript
export async function executeInWorkingDirectory<T>(
  workingDir: ContentNodeArray | string | undefined,
  env: Environment,
  execute: (resolvedPath?: string) => Promise<T>
): Promise<T> {
  const result = await resolveWorkingDirectory(workingDir, env);
  try {
    const path = result.type === 'path' ? result.path : undefined;
    return await execute(path);
  } finally {
    if (result.workspacePushed) {
      env.popActiveWorkspace();
    }
  }
}
```

## Examples
```mlld
var @dir = "/tmp"                              // String value
var @workspace = files ["task.md": @content]   // WorkspaceValue

exe @local = cmd:@dir { pwd }                  // Interpolates to string → filesystem path
exe @isolated = cmd:@workspace { pwd }         // Detected as WorkspaceValue → ShellSession

>> Parameters work the same way
exe @fn(ws) = cmd:@ws { cat task.md }
var @ws = files ["task.md": "content"]
run @fn(@ws)  // @ws resolves to WorkspaceValue, detected before interpolation
```

## Provider + cmd:@workspace precedence
When both provider and `cmd:@workspace` are specified, the provider takes precedence and the workspace reference is ignored. Commands execute in the provider environment, not local VFS. A warning is emitted if both are detected. This is consistent with existing provider semantics.

## Acceptance Criteria
- `resolveWorkingDirectory()` returns `WorkingDirectoryResult` discriminated union
- cmd:@workspace detects WorkspaceValue before interpolation
- cmd:@workspace pushes workspace onto stack, commands route through ShellSession
- cmd:@path still works for filesystem paths (no regression)
- exe parameters that are workspace values work: `exe @fn(ws) = cmd:@ws { ... }`
- Resolver-backed variables handled correctly (async resolution before type check)
- Provider overrides workspace when both specified (with warning)
- executeInWorkingDirectory() pops workspace in finally block
- Unit tests for pre-interpolation type checking (direct vars, resolver vars, string paths)
- Fixture tests for cmd:@workspace with workspace values
- Fixture tests for cmd:@path backward compatibility
