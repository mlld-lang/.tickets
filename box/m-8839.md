# file/files doc atoms are stubs — missing critical behavioral details

## Problem

`docs/src/atoms/output/05-file.md` is 3 lines of content. `docs/src/atoms/output/06-files.md` is 4 lines. These are the primary documentation for two new directives and they're missing nearly all critical information.

## Missing from file atom (05-file.md)

- What happens inside vs outside a box block (inside: writes to workspace VFS; outside: writes to real FS)
- What "relative" means (relative to project root / workspace root)
- Immutability: directive-level writes are one-shot, can't overwrite same path in same scope
- Error behavior for invalid paths

## Missing from files atom (06-files.md)

- Resolver creation: `files <@workspace/>` creates a VFS-backed resolver on first use — this is the primary mechanism for workspace creation and it's shown but never explained
- What the `<@resolver/>` angle-bracket-slash syntax means (named resolver target)
- Multiple writes to same resolver: subsequent `files <@workspace/subdir>` extends the same VFS
- `desc:` field: shown in example but not explained (populates `workspace.descriptions` map)
- Inside box behavior: writes to workspace VFS without creating a resolver
- git source: `files <@ws/> = git "url"` exists but is only documented in box-blocks atom, not here
- Workspace concept: no explanation of what a workspace IS (VFS + descriptions + lazy ShellSession)

## Cross-referencing gaps

- Neither atom links to or explains the box context
- git hydration is in box-blocks but not in files
- No atom explains the workspace concept as a first-class thing

## Fix

Expand both atoms following the docs style guide (docs/dev/DOCS-CLI.md): example-first, show output, present tense. Each should be at least as detailed as the box atoms. Rebuild LLM docs after: `mlld run llmstxt`.
