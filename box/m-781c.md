# file directive inside box writes to real FS instead of workspace VFS

## Reproduction

```mlld
box [
  file "src/main.js" = "console.log('main')"
  run cmd { cat src/main.js }
]
```

Produces ENOENT errors for the real path `/Users/.../src/main.js` (parent dir doesn't exist on real FS), then the ShellSession commands succeed because the VFS has the file. The `file` directive appears to write to BOTH real FS and VFS.

Flat paths (`file "hello.txt" = ...`) don't surface the error because the project root exists on the real FS, but the dual-write is still wrong.

## Expected

`file` inside a box block should write ONLY to the active workspace VFS via `executeWrite()` routing. No real filesystem write should be attempted.

## Root Cause (likely)

`executeWrite()` or the effect handler doesn't check for an active workspace before falling through to the real filesystem write path. The VFS write succeeds (ShellSession can read the file), but a parallel real FS write also fires and fails when parent dirs don't exist.

## Files to investigate

- `interpreter/eval/file.ts` — how evaluateFile calls executeWrite
- `interpreter/eval/write-executor.ts` — target routing logic, whether it checks active workspace
- The effect handler that intercepts file write effects

## QA context

Found in QA round 3. All other box features passing — anonymous box, resolver shorthand, nested boxes, cmd:@workspace, resolver reads, .mx.edits, workspace show serialization.
