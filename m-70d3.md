# [User Request] @mlld/pg and @mlld/sqlite modules for database-backed artifact storage

Create registry modules for database-backed output targets. The resolver system already supports custom output targets, so users should eventually be able to write:

```mlld
output @result to @db/artifacts/scan-123
append @finding to @db/findings
```

Each write should store the payload plus full label/provenance metadata (`@mx`) so artifact stores preserve security context, not just raw data.

Origin: Nexus team evaluation identified database-backed artifact storage as a gap. Their pipeline stores artifacts in a database, not files.

## Recommended implementation direction

Use mlld's Node-package import support directly instead of shelling out to CLIs.

This path appears viable because:

- mlld already documents direct Node package imports
- node import wrapping has fixture coverage for function wrapping, method binding, and constructors
- resolver write plumbing already exists in the interpreter

So the database client layer should be relatively straightforward if we pick stable libraries with simple APIs.

## Recommended packages

### `@mlld/pg`

Use `pg` (node-postgres).

Why:

- official and battle-tested Postgres client for Node
- built-in pooling via `Pool`
- clean async API with parameterized queries
- SSL/TLS support documented upstream

This is the safest first choice for Postgres-backed artifact storage.

### `@mlld/sqlite`

Use `better-sqlite3` for the first SQLite module.

Why:

- simple synchronous API that maps well to mlld wrappers
- explicit transaction support
- widely used package with prebuilt binaries for LTS Node versions
- much more ergonomic than callback-oriented `sqlite3`

`sqlite3` is a weaker first choice because its callback-heavy API is a worse fit for mlld's node import wrapping model.

Caveat: `better-sqlite3` is still a native addon. Since mlld is often used as a globally installed CLI, installation/binary compatibility is the main operational risk with this choice. If that friction proves too high in practice, fall back to a separate `@mlld/libsql` module or reconsider Node's built-in `node:sqlite` for a later slice.

## Optional follow-up / separate module

Consider `@libsql/client` later as a separate `@mlld/libsql` module rather than folding it into `@mlld/sqlite`.

Why not lead with it:

- it is a good fit for Turso/libSQL and can target remote or local/file URLs
- but it is a scoped package and a distinct product/storage model from plain SQLite files
- it should not complicate the first local-SQLite module

If direct scoped Node-package imports prove smooth in practice, this becomes an attractive second module.

## Scope note

There are really two layers here:

1. **Database client wrappers**
   Straightforward if implemented around `pg` and `better-sqlite3`.
2. **Resolver/prefix UX**
   Potentially slightly more work if we want modules to auto-provide `@db/...` style output targets out of the box.

The interpreter can already write through output resolvers, but module-driven resolver registration/prefix setup may still need a small amount of glue or config design.

## Suggested delivery order

1. Build `@mlld/pg` on top of `pg`
2. Build `@mlld/sqlite` on top of `better-sqlite3`
3. Verify provenance/labels storage shape (`payload`, `labels`, `provenance`, timestamps, path/key)
4. If resolver UX is blocked, ship callable storage helpers first and add `@db/...` output targets in the next slice

## Acceptance criteria

- `@mlld/pg` exists and can write/read artifact records with metadata using `pg`
- `@mlld/sqlite` exists and can write/read artifact records with metadata using `better-sqlite3`
- modules store payload plus label/provenance metadata
- at least one path to ergonomic artifact writes exists:
  - direct resolver output (`output ... to @db/...`), or
  - temporary callable helpers with clear follow-up ticket for resolver UX
