# Update box docs: VFS tool defaults, workspace resolver pattern, @toolbridge

## Context

The VFS MCP bridge feature (m-ce4b) is working but the docs are out of sync. Several gaps found during QA.

## Changes needed

### 1. Default VFS tools list is wrong
`config/12-box--config.md:44-46` and `config/10-box--basics.md:71` list default VFS tools as `["Bash", "Read", "Write"]`. Implementation is now `['Bash', 'Read', 'Write', 'Glob', 'Grep']`. Update all references.

### 2. Document the workspace resolver pattern for reading VFS files
The correct pattern for reading files from a box workspace:

```mlld
var @ws = box [
  file "data.txt" = "hello"
]
show <@ws/data.txt>    >> reads from workspace via resolver
```

Key points to document:
- `var @name = box [...]` binds the workspace to `@name`
- `<@name/file.txt>` reads from that workspace's VFS
- Inside the box body, use `run cmd { cat file.txt }` to read via ShellSession
- `<file.txt>` inside a box does NOT read from the active workspace — it reads from the real filesystem
- If the box returns a value via `=>`, the variable gets that value, not the workspace

### 3. Document @toolbridge return value and usage
`config/13-box--blocks.md:135` mentions @toolbridge as a one-liner. Document:
- Return value: `{ config, tools, mcpAllowedTools, inBox }`
- `config` — path to MCP config JSON (empty string if no MCP needed)
- `tools` — CSV of tool names (e.g., "Read,Glob")
- `mcpAllowedTools` — Claude CLI format names (e.g., "mcp__mlld_vfs__Read,mcp__mlld_vfs__Glob")
- `inBox` — boolean, true when active VFS bridge exists
- Must be called inside an exe invocation scope

### 4. Document when VFS defaults apply
The current docs don't explain the conditions. Document:
- VFS defaults apply when: box has a workspace AND no explicit config expression
- Explicit `with { tools: [...] }` overrides defaults
- Policy can further restrict via `policy.env.tools.allow/deny`

### 5. Clarify box naming patterns
- `var @name = box [...]` — workspace bound to @name, resolver access via <@name/path>
- `box @cfg [...]` — config-driven, @cfg is the config variable
- `box [...]` with `=> value` — anonymous, returns a value not the workspace (fine for side-effect boxes)
