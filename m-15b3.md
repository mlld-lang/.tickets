# claude -p hangs on MCP server init when spawned from mlld cmd directive

## Summary

`claude -p` with `--mcp-config` hangs indefinitely when spawned as a subprocess through mlld's `cmd` directive. The process enters the event loop but never starts the MCP server proxy process. The exact same command works instantly when run manually from the terminal.

## Relationship to #21099

GitHub issue anthropics/claude-code#21099 describes `claude -p` completing work but never exiting. This is the inverse: `claude -p` never starts work when given `--mcp-config`. Both are `claude -p` subprocess lifecycle bugs. The `@claudePoll` workaround in @mlld/claude was created specifically to sidestep #21099.

## Reproduction

The auto-bridge infrastructure works correctly — verified that `@mx.llm.config`, `@mx.llm.allowed`, `@mx.llm.inBox` all produce correct values. The generated MCP config is valid, the VFS socket is alive and accepts connections. The claude process simply never initiates MCP server startup.

### Works (manual, same command):
```bash
printf 'Say pong' | claude -p --model haiku --tools "" \
  --mcp-config "/tmp/mlld-toolbridge-call-config-XXXX.json" \
  --strict-mcp-config --allowedTools "mcp__mlld_vfs__Read"
# Returns "pong" immediately
```

### Hangs (spawned by mlld cmd directive):
```mlld
exe llm @test(prompt, config) = @prompt | cmd {
  claude -p --model haiku --tools "" --mcp-config "..." --strict-mcp-config --allowedTools "..."
}
```
Process tree shows:
- mlld spawns `/bin/sh -c printf '%s' '...' | claude -p ...`
- sh spawns `claude -p` (the actual binary)
- `claude -p` has NO child processes (never starts MCP proxy)
- `claude -p` sits in `kevent64` / `__psynch_cvwait` (event loop idle)

### Also works (no MCP config):
```mlld
exe llm @test(prompt) = @prompt | cmd { claude -p --model haiku }
```
This returns "pong" in ~15 seconds. Adding `--mcp-config` causes infinite hang.

## Diagnostic evidence

1. **Socket test**: VFS unix socket accepts connections (verified with `net.createConnection`)
2. **Process sample**: `claude -p` (PID) main thread in `kevent64` (I/O wait), worker threads in `__psynch_cvwait`. Normal idle Node.js pattern — it's waiting but never progresses.
3. **No children**: `claude -p` never spawns the MCP proxy child process defined in the config
4. **Config valid**: MCP config JSON is well-formed, proxy .cjs script exists, node binary path valid
5. **spy ls**: No active claude session created — claude never reaches the point of making an API call
6. **Claude version**: 2.1.32 (native arm64 binary, macOS)

## Likely cause

Something about the process spawning context (environment, file descriptors, process group, signal handling) when `claude -p` is spawned as a grandchild of a Node.js process through `/bin/sh -c` prevents it from initializing MCP servers. The same binary with the same arguments works when spawned directly from a terminal shell.

## Impact

Blocks all VFS + tool bridging QA tests. The auto-bridge code (`@mx.llm`) is working correctly but can't be validated end-to-end because `claude -p` hangs when given the generated MCP config.

## Workaround options

1. Use `@claudePoll` pattern (fire-and-forget + marker file polling) — already exists for #21099
2. Investigate whether specific env vars or fd inheritance fixes the spawn behavior
3. Test with newer Claude CLI versions (current: 2.1.32)


## Notes

**2026-03-06T04:11:19Z** ## Investigation update

### What we tried

**1. Module-level fix: switch @claude from `cmd` to `sh` with file-backed stdin**
- Rationale: `sh` with `< "$PROMPT_FILE"` gives claude a file fd for stdin instead of a pipe fd
- Result outside box: works perfectly — `@claude("Say pong", { model: "haiku" })` returns immediately
- Result inside box: fails — `mktemp: command not found`, `claude: command not found`, `cd: Not a directory`
- Root cause: inside a box, `sh` routes through ShellSession/BashExecutor over VirtualFS (BashExecutor.ts:244, ShellSession.ts:71), not a host shell. Host binaries don't exist by design. PATH changes won't help.
- Key insight from GPT5.4: `cmd` has a special llm escape hatch — CommandExecutorFactory.ts:104 checks for a workspace, and :233 routes `exe llm` commands to the real host ShellCommandExecutor. That's why `cmd` can see real `claude` while `sh` cannot.

**2. Runtime fix: spawn instead of exec in ShellCommandExecutor**
- Changed `executeBufferedCommand()` to use `spawn()` with direct `stdin.write()` + `stdin.end()` when `options.input` is provided, instead of `exec()` with `printf '%s' '...' | command`
- All 5173 automated tests pass
- Process tree confirms: no more `printf | claude` — command is now `/bin/sh -c claude -p ...` with stdin written directly by Node.js
- Result: **still hangs**. Claude process enters event loop, never starts MCP proxy child, no spy session created. Identical behavior to the exec path.

**3. Manual reproduction paradox**
- The exact same `claude -p` command with the exact same MCP config file works instantly when run from a terminal
- The VFS socket accepts connections (verified with `net.createConnection`)
- The MCP config JSON, proxy .cjs script, and node binary path are all valid
- Claude version: 2.1.32 (native arm64 binary)

### Current understanding

The hang is NOT caused by:
- ❌ pipe-backed stdin (spawn with direct write also hangs)
- ❌ the `printf | command` pattern specifically
- ❌ `child_process.exec` vs `child_process.spawn` (both hang)
- ❌ MCP config validity (works manually)
- ❌ VFS socket readiness (accepts connections)

The hang IS caused by:
- ✅ Something about the process spawning context when `claude -p` is a grandchild of a Node.js process
- ✅ Both `exec()` and `spawn({ shell: true })` go through `/bin/sh -c`, which adds a shell layer
- ✅ Only occurs when `--mcp-config` is passed — same command without MCP config works fine through both paths

### Untried

- `spawn()` with `shell: false` and explicit argv (no `/bin/sh -c` wrapper at all)
- Environment variable differences between terminal and Node.js subprocess context
- File descriptor inheritance (`stdio` options, detached mode)
- Testing with newer Claude CLI versions (current: 2.1.32)
- Checking if Claude CLI inspects its parent process tree or TTY state during MCP init

**2026-03-06T05:04:38Z** ## Resolution

This was not ultimately a Claude MCP spawn-context bug.

The actual root cause was malformed argv caused by nested quoted command-fragment interpolation in `@claude`.

`modules/claude/index.mld` built a quoted flag string via `@claudeFlags(...)` and then interpolated that string into another `cmd { ... }` template. In the `cmd` direct-spawn path, that produced broken argv splitting for quoted values.

Concrete proof from a debug helper:
- direct inline quoted args inside `cmd { ... }` arrived correctly
- nested fragment interpolation did not
- the helper received `prompt = "Say` instead of the full prompt
- the helper received the intended log path as `just`, and wrote its logfile into a file literally named `just` in the repo root

That same bad argv construction affected the Claude invocation, including `--mcp-config` and `--allowedTools`, which made the failure look like an MCP startup hang.

Actual fix:
- inline the Claude flags directly in `@claude(...)` instead of interpolating a prebuilt quoted `@flags` fragment
- keep the MCP bridge notification fix as well: bridge servers must not reply to `notifications/initialized`

Verification:
- `node ./bin/mlld-wrapper.cjs tmp/qa-box-claude-when.mld.md --no-checkpoint`
- result after fix: exits cleanly with `pong` in ~3.6s

Implication:
- nested prebuilt shell/command fragments are unsafe in `cmd { ... }`
- if a command needs reusable argument composition, mlld should eventually support first-class argv/vector composition rather than string fragment reuse
